# 工程架构决策

## 1. 参考来源

本项目参考 `D:\desk\tjupro\reg-pilot` 的工程组织方式。

可复用的规范：

- Monorepo 结构。
- `apps/server` 放 Spring Boot 后端。
- `apps/web` 放 Vue 3 前端。
- 后端统一 `/api` 前缀。
- 后端统一返回结构 `Result<T>`。
- 后端使用 `controller`、`service`、`mapper`、`entity`、`param`、`vo`、`config`、`common` 分层。
- 前端使用 `api/http.ts` 封装 Axios。
- 前端按 `views`、`components`、`router`、`stores`、`api/modules` 组织代码。
- 配置分为 `application.yml` 和 `application-dev.yml`，敏感值使用环境变量覆盖。

不直接照搬的部分：

- 不做独立 Python agent 服务。
- 不引入 MinIO。
- 不做 CAS、RBAC、复杂权限系统。
- 不做复杂后台管理端。
- 不做三端联动，只做前端 + 后端两端。

## 2. 最终技术栈

### 后端

- Java 17
- Spring Boot 3.3.6
- Maven
- MyBatis-Plus 3.5.9
- MySQL 8
- Lombok
- Jackson
- Spring Validation
- Spring Boot Test

Java 版本定为 17。

原因：

- Spring Boot 3 支持 Java 17，兼容性更稳。
- `reg-pilot` 文档和徽章标注 Java 17。
- 本机 `java -version` 默认是 Java 17。
- Maven 虽然可用 JDK 21 运行，但项目编译目标应显式固定为 17，减少部署不一致风险。

执行约定：

- 本地可以用 JDK 21 运行 Maven。
- 项目 `pom.xml` 中仍显式配置 `java.version=17` 和 compiler `release=17`。
- 不使用 Java 21 专属语法，避免换机器或部署时出问题。
- 如果后期确认部署环境和评审环境都稳定支持 JDK 21，再考虑升级。

### 前端

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios
- Element Plus

采用 Element Plus 是为了快速完成可用、规整的表单、按钮、卡片和提示，不把时间浪费在基础组件上。

### 数据库

- MySQL 8

第一阶段只做必要表：

- 题卡表
- 面试会话表
- 回答记录表
- 复盘报告表

如果一周 MVP 时间紧，题卡可以先用 JSON 文件，训练会话和报告再落 MySQL。

## 3. 仓库结构

采用 Monorepo：

```text
agent2026-interview-agent/
├── apps/
│   ├── server/          # Spring Boot 后端
│   └── web/             # Vue 3 前端
├── docs/                # 中文设计、任务、周报文档
├── screenshots/         # 演示截图或视频
├── DESIGN.md            # 根目录技术设计文档
├── README.md
├── .env.example
└── .gitignore
```

暂不创建 `apps/agent`。本项目的大模型调用由 Spring Boot 后端直接代理学校 `tju-llm` API。

这里的取舍是：逻辑上保留 agent 能力，部署上不单独拆 agent 端。

Spring Boot 内部可以有这些 agent 编排类：

- `InterviewPlanner`
- `QuestionEngine`
- `AnswerEvaluator`
- `FollowUpDecider`
- `ReportGenerator`
- `TjuLlmClient`

它们共同构成面试智能体，但仍在 `apps/server` 内运行。

## 4. 后端包结构

基础包名：

```text
com.agent2026.interview
```

建议结构：

```text
apps/server/src/main/java/com/agent2026/interview/
├── InterviewAgentApplication.java
├── controller/
├── service/
│   └── impl/
├── mapper/
├── entity/
├── param/
├── dto/
├── vo/
├── client/
├── config/
├── common/
└── enums/
```

职责说明：

- `controller`：HTTP 接口层。
- `service`：业务接口。
- `service.impl`：业务实现。
- `mapper`：MyBatis-Plus 数据访问。
- `entity`：数据库实体。
- `param`：前端请求参数。
- `dto`：内部数据传输对象。
- `vo`：返回给前端的视图对象。
- `client`：外部服务客户端，如 `TjuLlmClient`。
- `config`：配置类、异常处理、跨域等。
- `common`：统一返回、异常、错误码。
- `enums`：训练模式、模块、难度、会话状态等枚举。

## 6. 配置规范

后端配置：

```text
apps/server/src/main/resources/
├── application.yml
└── application-dev.yml
```

`application.yml` 放通用配置。  
`application-dev.yml` 放开发环境配置。  
敏感值从环境变量读取。

学校 API 配置：

```yaml
tju:
  llm:
    api-url: ${TJU_LLM_API_URL:}
    api-key: ${TJU_LLM_API_KEY:}
    model: ${TJU_LLM_MODEL:tju-llm}
```

不得在配置文件中写真实 API Key。

## 8. 架构取舍

### 为什么不单独做 Agent 服务

一周 MVP 目标下，单独维护 Python agent 服务会增加部署和联调成本。  
本项目的智能体逻辑可以先放在 Spring Boot 后端中：

- 题卡选择
- 面试状态
- 追问决策
- 复盘生成
- 学校 API 调用

后续如果系统复杂，再考虑拆出独立 agent 服务。

判断是否需要独立 agent 端的标准：

- 需要 RAG、向量库或大规模题库检索。
- 需要复杂多智能体工作流。
- 需要独立扩展模型服务吞吐。
- 需要 Python 生态中的 ASR、NLP 或评测工具。
- Spring Boot 内的 prompt 编排已经明显难以维护。

在当前比赛 MVP 中，这些条件都不成立，所以不拆。

### 为什么使用 MyBatis-Plus

参考 `reg-pilot`，MyBatis-Plus 适合快速完成 CRUD 和数据库映射。  
本项目的表结构不复杂，MyBatis-Plus 足够。

### 为什么使用 Element Plus

比赛更看重功能闭环和智能体能力。Element Plus 能快速提供稳定 UI 组件，减少前端实现成本。
