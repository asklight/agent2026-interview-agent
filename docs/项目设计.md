# 北洋 Java 面试官项目设计

## 1. 项目定位

项目建议仓库名：`agent2026-interview-agent`

项目建议中文名：`北洋 Java 面试官`

本项目面向天津大学准备软件岗就业、实习、秋招和春招的学生，提供 Java 后端技术面试训练。它不是普通聊天机器人，也不是简单题库，而是一个能够模拟真实技术面试流程的 AI 智能体。

核心目标：

- 模拟 Java 后端技术一面。
- 围绕回答动态追问。
- 针对项目经历进行深挖。
- 训练算法思路表达。
- 生成结构化复盘报告。

## 2. 比赛规则适配

技术实现赛道重点看作品能否稳定运行、是否有真实校园应用价值、是否有工程实现质量、是否适配比赛平台。

本项目需要注意：

- 仓库名使用 `agent2026-` 前缀。
- 所有大模型请求走比赛分配的 `tju-llm` 专属 API 地址。
- API Key 只能放在后端环境变量中，不能提交到 Git 仓库，也不能写在前端代码里。
- 题库、示例项目和说明文案需要原创整理，避免直接复制商业题库或培训材料。
- 简历和项目经历输入需要提示用户删除手机号、邮箱、身份证号等敏感信息。
- 如果做语音输入，不保存原始音频，只保存用户确认提交后的文字。
- 提交材料需要包含 `README.md`、`DESIGN.md`、演示截图或视频、版权合规承诺书。
- 需要在 2026 年 9 月 1 日前尽量做出可体验版本，方便全校用户评价。
- 最终提交截止时间是 2026 年 9 月 30 日 22:00。

代码提交要求：

- 学校官方提交平台是天津大学 GitLab：`gitlab.tju.edu.cn`。
- GitLab 项目名称必须使用 `agent2026-` 前缀。
- 创建项目时可先设为私有，开发完成后应改为内部可见，方便校内用户查看、评价和收藏。
- GitHub 可以作为个人备份、镜像仓库或对外展示仓库，但不能替代学校 GitLab。
- 如果同时使用 GitLab 和 GitHub，必须以学校 GitLab 为主仓库，最终提交材料和比赛统计都以学校平台为准。
- 不论 GitLab 还是 GitHub，都不能提交 API Key、`.env` 文件或其他敏感配置。

## 3. 目标用户

第一版只服务 Java 后端软件岗学生。

主要用户：

- 准备 Java 后端实习的本科生。
- 准备秋招、春招的本科生或研究生。
- 需要练习 Java 后端八股、项目表达和算法口述的同学。

第一版暂不覆盖：

- 前端、测试、算法岗等其他岗位。
- 完整 PDF 简历解析。
- 在线算法判题系统。
- 实时语音通话式面试。
- 排行榜、社区、社交功能。

## 4. 核心功能

### 4.1 Java 核心知识面试

这是原来“八股追问”的正式名称。

目标是训练 Java 后端高频核心知识。系统不是简单给题和标准答案，而是按真实面试方式提问、追问、评价。

首发模块：

- Java 基础与集合
- JVM
- Java 并发
- MySQL
- Redis
- Spring / Spring Boot
- 计算机网络
- 操作系统

训练强度：

- 快速查漏：每题最多一次追问，适合短时间刷题。
- 标准面试：每题两到三次追问，适合模拟真实一面。
- 深度拷问：围绕一个知识点连续追问，适合专项突破。

### 4.2 项目经历深挖

用户粘贴自己的项目经历或简历片段，系统根据技术栈和个人职责生成追问。

输入内容建议包括：

- 项目名称
- 技术栈
- 个人职责
- 核心功能
- 性能、稳定性或工程亮点

示例：

> 基于 Spring Boot、MySQL 和 Redis 的校园二手交易平台，负责商品发布、搜索、登录鉴权和热门商品缓存。

系统可追问：

- 为什么这个项目需要 Redis？
- 缓存 key 是怎么设计的？
- 缓存和数据库如何保持一致？
- 登录鉴权是怎么做的？
- 如果并发量上来，系统瓶颈在哪里？
- 这个项目中哪些部分是你独立完成的？

### 4.3 算法思维面试

第一版不做完整在线判题系统，只做面试口述型算法训练。

训练内容：

- 让用户说明解题思路。
- 追问时间复杂度和空间复杂度。
- 追问边界情况。
- 允许用户粘贴 Java 代码或伪代码。
- AI 对思路、复杂度、边界处理和代码表达进行点评。

不做的内容：

- 不执行用户提交的代码。
- 不做完整 OJ。
- 不追求大规模算法题库。

### 4.4 综合模拟面试

综合模拟面试把项目经历、Java 核心知识和算法思维组合成一场完整 Java 技术一面。

建议流程：

- 自我介绍或项目切入
- 项目经历深挖
- Java 后端核心知识提问
- 一道算法思路题
- 面试总结和复盘报告

### 4.5 语音转文字答题

语音输入用于提升真实面试感，但不能影响主流程稳定。

第一版设计：

- 默认支持文字输入。
- 可选支持麦克风口述。
- 系统把口述转成文字后展示给用户。
- 用户确认或修改后再提交。
- AI 只基于确认后的文字进行评分和追问。

### 4.6 结构化复盘报告

所有训练模式结束后都生成复盘报告。

报告包含：

- 总分
- 各模块分数
- 答得好的点
- 漏掉的关键点
- 错误表达或面试风险点
- 推荐下一轮训练模块
- 三道补强练习题

## 5. 模块边界

Java 核心知识面试和项目经历深挖会有交叉，但边界不同。

Java 核心知识面试：

- 从标准知识点出发。
- 目标是检查基础是否扎实。
- 问题更标准、更模块化。

项目经历深挖：

- 从用户自己的项目经历出发。
- 目标是检查项目真实性、技术理解、方案取舍和表达能力。
- 问题更个性化、更场景化。

两者可以复用同一套追问和评分引擎，但产品入口和评估重点要分开。

## 6. 技术架构

建议做成 Web 应用。

确定技术栈：

- 前端：Vue 3 + TypeScript
- 后端：Spring Boot
- 数据库：MySQL
- 大模型：比赛专属 `tju-llm` API

这样选择的原因：

- 项目主题是 Java 后端面试，用 Spring Boot 实现更贴合方向。
- Vue 适合快速完成可用界面，方便展示训练流程。
- MySQL 本身也是 Java 后端面试高频模块，项目实现和面试内容能形成呼应。
- 后端统一代理模型请求，可以避免 API Key 暴露在前端。

后端模块：

- Interview Planner：生成本场面试计划。
- Question Engine：负责选题和追问。
- Answer Evaluator：负责回答评分。
- Follow-up Decider：判断是否追问、提示、换题或结束。
- Report Generator：生成复盘报告。
- Session Manager：保存会话状态。
- LLM Client：统一封装比赛 API 调用、错误处理和限速逻辑。

## 7. 题卡设计

第一版不要完全依赖 AI 临时编题，而是准备结构化题卡。

题卡字段建议：

- 题目 ID
- 所属模块
- 难度
- 主问题
- 核心答案要点
- 常见错误
- 追问问题
- 场景追问
- 评分标准

示例：

```json
{
  "id": "redis-cache-consistency",
  "module": "Redis",
  "difficulty": "medium",
  "main_question": "缓存和数据库如何保持一致？",
  "key_points": ["更新数据库后删除缓存", "延迟双删", "binlog/Canal", "最终一致性"],
  "common_mistakes": ["先更新缓存再更新数据库", "认为强一致性总是必要"],
  "followups": ["为什么不推荐先删缓存再更新数据库？", "延迟双删有什么问题？"],
  "scenario_followups": ["高并发商品详情页如何避免缓存击穿？"],
  "scoring_rubric": ["方案准确性", "场景意识", "一致性理解", "风险表达"]
}
```

## 8. 一次面试的数据流

1. 用户选择训练模式。
2. 用户选择模块、难度和训练时长。
3. 如果是项目经历深挖，用户粘贴项目描述。
4. 后端生成面试计划。
5. 系统提出第一个问题。
6. 用户输入或口述回答。
7. 后端调用 `tju-llm` 进行评分和分析。
8. 系统决定追问、提示、换题或结束。
9. 面试结束后生成复盘报告。

模型输出尽量要求结构化 JSON，方便前端稳定展示分数、遗漏点和下一步动作。

## 9. 评分标准

通用评分维度：

- 正确性
- 完整性
- 深度
- 表达结构
- 场景意识

项目经历深挖额外评分：

- 个人职责是否清楚
- 技术取舍是否合理
- 项目理解是否真实

算法思维面试额外评分：

- 问题拆解能力
- 复杂度分析
- 边界情况处理
- Java 表达能力

## 10. 成功标准

项目达到参赛要求的标准：

- 用户可以完成一场 Java 后端模拟面试。
- 系统能根据回答进行有意义的追问。
- 复盘报告具体，不是空泛评价。
- 项目经历深挖能围绕用户输入内容提问。
- API 调用走比赛专属地址。
- 代码和文档结构清晰。
- 提交材料完整合规。
