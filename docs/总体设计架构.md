# 总体设计架构

## 1. 总体目标

本项目是面向 Java 后端就业实习面试训练的 AI 智能体系统。

系统核心目标：

- 让用户完成 Java 后端模拟面试。
- 根据用户回答动态追问。
- 使用结构化题卡保证训练稳定性。
- 使用 Milvus 检索八股和面经资料增强追问与复盘。
- 使用学校 `tju-llm` API 完成提问、评价和复盘生成。

## 3. 应用划分

### 3.1 apps/web

前端应用，负责用户交互。

技术栈：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios
- Element Plus

### 3.2 apps/server

后端主服务，负责业务编排、数据库读写、Milvus 检索和学校 API 调用。

技术栈：

- Java 17
- Spring Boot 3.3.6
- Maven
- MyBatis-Plus
- MySQL
- Milvus Java/REST Client

### 3.3 apps/ingest

离线灌库工具，负责把八股资料、面经和项目追问案例清洗后写入 Milvus。

建议技术：

- Python
- sentence-transformers 或 FlagEmbedding
- Milvus Python SDK

注意：`apps/ingest` 不是线上服务，不参与用户实时面试请求。

职责：

- 管理结构化题卡。
- 按模块、难度、标签选题。

核心类：

- `QuestionCardService`
- `QuestionSelector`
- `QuestionCard`
- `QuestionCardMapper`

### 5.6 interview 模块

职责：

- 管理一次面试训练流程。

核心类：

- `InterviewPlanner`
- `InterviewSessionService`
- `InterviewState`
- `QuestionEngine`

`InterviewPlanner`：

- 根据训练模式、模块和难度生成面试计划。

`QuestionEngine`：

- 从题卡中选择当前问题。
- 结合已问历史避免重复。

`InterviewState`：

- 保存当前会话状态。
- 保存已问问题、当前题目、追问次数、薄弱点。

### 5.7 evaluation 模块

职责：

- 评价用户回答。
- 判断漏掉哪些关键点。

核心类：

- `AnswerEvaluator`
- `EvaluationPromptBuilder`
- `AnswerEvaluationResult`

输入：

- 当前题卡
- 用户回答
- Milvus 召回片段
- 会话上下文

输出：

- 分数
- 命中要点
- 遗漏要点
- 错误点
- 建议追问方向

### 5.8 followup 模块

职责：

- 决定下一步动作。

核心类：

- `FollowUpDecider`
- `FollowUpPromptBuilder`
- `NextAction`

- `ASK_FOLLOW_UP`
- `GIVE_HINT`
- `NEXT_QUESTION`
- `FINISH_SESSION`

### 5.9 retrieval 模块

核心类：

- `VectorRetrievalService`
- `KnowledgeChunk`
- `RetrievalQueryBuilder`

检索输入：

- 当前模块

- 标签

检索输出：

- topK 知识片段
- 来源标题
- 相似度
- 标签

### 5.10 report 模块

## 10. 核心流程

### 10.1 Java 核心知识训练

```text
用户选择模块
-> 创建 session
-> QuestionSelector 选题
-> 返回主问题
-> 用户回答
-> VectorRetrievalService 检索相关资料
-> AnswerEvaluator 调用 tju-llm 评价
-> FollowUpDecider 决定追问或下一题
-> 训练结束
-> ReportGenerator 生成报告
```

### 10.2 项目经历深挖

```text
用户输入项目经历
-> 提取技术栈
-> Milvus 检索相似面经和项目追问
-> LLM 生成项目追问
-> 用户回答
-> LLM 评价项目理解和表达
-> 写入复盘报告
```

### 10.3 复盘报告生成

```text
读取 session
-> 读取所有 answer
-> 汇总评分和薄弱点
-> Milvus 检索推荐资料
-> LLM 生成复盘文本
-> 保存 interview_report
```

### 13.4 表设计取舍

第一周不应该一次性建完所有表。

建议顺序：

```text
第一周：question_card / interview_session / interview_answer / interview_report
第二阶段：knowledge_source / knowledge_document / knowledge_chunk / ingest_job
第三阶段：project_profile / weak_point / practice_plan / feedback
```

这样既能保证架构完整，又不会在一开始被数据库设计拖住。

## 16. 为什么现在文档不一次写成 reg-pilot 那么大

`reg-pilot` 是完整平台，所以表和接口非常多。  
本项目当前目标是比赛作品和一周可演示 MVP，需要先做核心闭环。

正确做法是：

```text
完整架构先规划清楚
实现时按 P0/P1/P2 分阶段落地
```

否则一开始照着完整平台建几十张表和几十个接口，会导致主线拖慢。

## 17. 设计原则

- 先保证结构化题卡流程稳定。
- Milvus 做增强，不做唯一依赖。
- tju-llm 是唯一运行时大模型。
- API Key 只在后端环境变量中。
- 前端不直接调用模型。
- 每个模块有明确输入和输出。
- 一周 MVP 先做能跑通的闭环。
