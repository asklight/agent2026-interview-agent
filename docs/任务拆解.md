# 任务拆解

## 1. 总目标

在技术实现赛道截止前，完成一个可运行、可演示、可被同学体验的 Java 后端面试训练智能体。

最小可用闭环：

> 用户选择训练模式 -> 系统提出问题 -> 用户回答 -> 系统追问或评价 -> 生成复盘报告

第一版不要追求功能全，而要保证这个闭环稳定、清晰、有价值。

## 2. 优先级说明

- P0：必须完成，否则项目不能算可用。
- P1：重要增强，能明显提高参赛效果。
- P2：锦上添花，时间足够再做。

## 3. 阶段 0：规则与项目管理

优先级：P0

目标：

- 确保项目符合学校技术实现赛道要求。
- 避免后期因为材料、仓库、API、合规问题扣分。

任务：

- 确认学校 GitLab 仓库名为 `agent2026-interview-agent`。
- 确认学校 GitLab 使用 `main` 分支。
- 确认 GitHub 只作为备份仓库。
- 确认 `.env` 不会提交。
- 准备根目录 `README.md`。
- 准备根目录 `DESIGN.md`。
- 准备 `screenshots/` 目录。
- 下载并放入版权合规承诺书。

验收标准：

- `git status` 干净。
- `git push` 能推送学校 GitLab。
- `git push github` 能同步 GitHub。
- 仓库中没有 API Key。

## 4. 阶段 1：工程骨架

优先级：P0

目标：

建立 Spring Boot + Vue 3 + MySQL 的基础工程。

后端任务：

- 初始化 Spring Boot 项目。
- 使用 Maven 管理依赖。
- 配置 Java 17。
- 增加基础包结构。
- 增加健康检查接口：`GET /api/health`。
- 增加统一返回结构。
- 增加全局异常处理。

前端任务：

- 初始化 Vue 3 + TypeScript 项目。
- 建立基础路由。
- 建立基础布局。
- 增加首页入口。
- 调用后端健康检查接口。

数据库任务：

- 创建 MySQL 数据库。
- 配置本地数据库连接。
- 确认后端能连接数据库。

验收标准：

- 后端能启动。
- 前端能启动。
- 前端能请求后端 `GET /api/health`。
- 后端能连接 MySQL。

## 5. 阶段 2：比赛 API 调用

优先级：P0

目标：

让后端安全调用学校 `tju-llm` API。

任务：

- 新增 `LLMClient`。
- 从环境变量读取 `TJU_LLM_API_URL`、`TJU_LLM_API_KEY`、`TJU_LLM_MODEL`。
- 实现普通非流式调用。
- 处理 401、429、500 等错误。
- 增加简单测试接口：`POST /api/llm/test`。
- 前端增加 API 测试按钮。

验收标准：

- 后端能调用学校 API。
- API Key 不出现在日志、前端、Git 仓库中。
- 429 时返回明确提示。
- 401 时提示检查 Key。

## 6. 阶段 3：题卡系统

优先级：P0

目标：

建立 Java 核心知识面试的题目基础。

任务：

- 设计题卡 JSON 结构。
- 准备 Java、MySQL、Redis、Spring 四个模块。
- 每个模块至少 5 张题卡。
- 每张题卡包含主问题、答案要点、常见错误、追问、评分点。
- 后端能读取题卡。
- 前端能展示题卡模块列表。

验收标准：

- 至少 20 张题卡可用。
- 系统能按模块随机抽题。
- 每道题都有追问和评分标准。

## 7. 阶段 4：Java 核心知识面试

优先级：P0

目标：

完成第一个真正可用的训练模式。

任务：

- 用户选择模块和难度。
- 系统生成本轮训练计划。
- 系统每次展示一个问题。
- 用户输入回答。
- 后端调用 LLM 对回答进行评价。
- 系统根据评价决定追问或换题。
- 保存本轮会话状态。

验收标准：

- 用户能完成一轮 Java 核心知识训练。
- 系统能进行至少一次有效追问。
- 不是简单“出题 + 标准答案”，而是有面试感。

## 8. 阶段 5：回答评分与追问决策

优先级：P0

目标：

让系统具备稳定的智能体行为。

任务：

- 设计评分维度：正确性、完整性、深度、表达结构、场景意识。
- 要求 LLM 输出结构化 JSON。
- 根据评分决定下一步动作。
- 支持三种动作：继续追问、给提示、换下一题。
- 限制单题最大追问次数，避免陷入循环。

验收标准：

- 回答浅时系统能追问。
- 回答错误时系统能指出问题。
- 回答较好时系统能进入下一题。
- 输出结构能被前端稳定解析。

## 9. 阶段 6：复盘报告

优先级：P0

目标：

每次训练结束后生成结构化复盘。

任务：

- 统计本轮问题和回答。
- 汇总模块得分。
- 汇总漏掉的关键点。
- 生成下一轮训练建议。
- 前端展示报告页面。

验收标准：

- 报告包含总分、模块分数、薄弱点、补强建议。
- 报告内容具体，不是空泛评价。
- 用户能根据报告知道下一步练什么。

## 10. 阶段 7：项目经历深挖

优先级：P1

目标：

让系统根据用户项目经历进行个性化追问。

任务：

- 前端提供项目经历输入框。
- 提示用户删除敏感信息。
- 后端提取项目技术栈、个人职责和功能点。
- 根据项目内容生成追问。
- 项目深挖结果进入复盘报告。

验收标准：

- 用户粘贴项目描述后，系统能围绕项目内容提问。
- 问题与项目技术栈相关。
- 能追问个人职责和技术取舍。

## 11. 阶段 8：综合模拟面试

优先级：P1

目标：

组合 Java 核心知识、项目深挖和算法思维，形成完整模拟面试。

任务：

- 设计综合面试流程。
- 支持项目切入。
- 支持 Java 核心知识问题。
- 支持一道算法思路题。
- 统一生成复盘报告。

验收标准：

- 用户能完成一场 10 到 15 分钟模拟技术一面。
- 面试内容包含项目、知识点和算法。
- 报告能分别评价三个部分。

## 12. 阶段 9：算法思维面试

优先级：P1

目标：

训练 Java 后端面试中的算法口述能力。

任务：

- 准备常见算法题。
- 支持用户描述思路。
- 支持用户粘贴 Java 代码或伪代码。
- 评价复杂度、边界情况和代码表达。

验收标准：

- 系统能完成一道算法题的口述训练。
- 不执行用户代码。
- 不做在线判题。

## 13. 阶段 10：语音转文字答题

优先级：P2

目标：

提升面试真实感，但不影响主流程。

任务：

- 前端增加麦克风按钮。
- 把口述内容转成文字。
- 用户确认后再提交。
- 保留文字输入兜底。

验收标准：

- 语音失败时不影响文字答题。
- 不保存原始音频。

## 14. 阶段 11：体验与稳定性

优先级：P1

目标：

让项目适合给同学试用。

任务：

- 优化页面布局。
- 增加加载状态。
- 增加错误提示。
- 增加训练历史。
- 处理 API 超时。
- 处理 429 限速。

验收标准：

- 普通用户可以无说明完成一次训练。
- API 错误不会导致页面崩溃。
- 复盘报告能正常保存和查看。

## 15. 阶段 12：提交材料

优先级：P0

目标：

满足学校最终提交要求。

任务：

- 更新 `README.md`。
- 编写根目录 `DESIGN.md`。
- 准备 `screenshots/`。
- 录制演示视频。
- 放入版权合规承诺书。
- 检查仓库可见性。
- 检查 API Key 泄露。

验收标准：

- 仓库材料完整。
- 学校 GitLab 为内部可见。
- 演示视频能体现核心智能体能力。
- 2026-09-30 22:00 前完成提交。

- 前端能请求后端。

- 用户能粘贴项目经历。

## 20. 范围裁剪规则

如果进度落后，按下面顺序砍范围。

第一批可以砍：

- 语音转文字。
- 训练历史。
- 算法专项面试。
- 页面动效。
- 多主题 UI。

第二批可以砍：

- 操作系统模块。
- 计算机网络模块。
- 深度拷问模式。
- 复杂图表报告。

不能砍：

- Java 核心知识面试。
- 学校 API 调用。
- 追问能力。
- 复盘报告。
- README、DESIGN、截图/视频、版权承诺书。

最低可提交版本：

- Spring Boot + Vue 能运行。
- 用户能做 Java 核心知识训练。
- 系统能追问。
- 系统能生成复盘报告。
- 所有模型请求走学校 API。

## 21. 开发工作习惯

每天结束前：

- 提交一次代码，或者明确说明为什么不能提交。
- 记录当天卡点。
- 确认没有把 `.env` 或 API Key 加入 Git。

每周结束前：

- 填写一份周报。
- 跑一遍核心流程。
- 更新下周任务。
- 检查比赛规则相关材料是否缺失。

Commit 建议：

- `docs: update task breakdown`
- `backend: add health endpoint`
- `frontend: initialize vue app`
- `backend: add tju llm client`
- `feature: add java knowledge interview flow`
- `feature: add review report`

分支策略：

- 学校 GitLab 使用 `main`。
- GitHub 使用 `master`。
- 本地开发默认在 `main`。
- 推学校：`git push`
- 推 GitHub：`git push github`
