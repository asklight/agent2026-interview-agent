# 项目交接与运维手册

本文档用于模型切换、同学接手、服务器运维和比赛提交前复查。

目标是让接手者先读完本文档，再结合 README 和设计文档继续开发，不需要重新追问基础背景。

## 1. 必读顺序

建议接手者按下面顺序阅读：

1. `README.md`：项目定位、比赛要求、技术栈、本地运行入口。
2. `docs/项目交接与运维手册.md`：当前真实状态、部署方式、排查命令和注意事项。
3. `deploy/README.md`：GitHub Actions、Docker Compose、生产环境变量和 VPN sidecar 细节。
4. `docs/总体设计架构.md`：完整模块、数据库表、API 和后续规划。
5. `docs/工程架构决策.md`：Java 版本、Monorepo、Spring Boot/Vue/MySQL/Milvus 的决策依据。
6. `docs/任务拆解.md` 和 `docs/一周冲刺计划.md`：后续开发任务和阶段验收标准。

如果只想快速接手部署和排查，优先读本文档和 `deploy/README.md`。

## 3. 仓库与分支规则

本地开发主分支：

```bash

```

学校 GitLab 是比赛正式提交仓库：

```bash

branch: main
```

GitHub 是备份和自动部署仓库：

```bash
remote: github
branch: master
```

常用推送：

```bash
git push origin main
git push github
```

当前 `github` 远端已配置 push refspec，本地执行：

```bash
git push github
```

等价于：

```bash
git push github main:master
```

注意：

- 学校 GitLab 不能被 GitHub 替代。
- 比赛统计、提交和评审应以学校 GitLab 为准。
- GitHub `master` 推送会触发 GitHub Actions 自动部署。
- 不要提交 `.env`、`.env.prod`、API Key、数据库密码、VPN 账号密码、服务器密码或 SSH 私钥。

## 5. 本地开发运行

推荐环境：

- JDK 17
- Maven 3.9+
- Node.js 20+
- MySQL 8.0

本地根目录需要创建 `.env`，参考：

```text
.env.example
```

必须配置但不要提交：

```env
MYSQL_URL=...
MYSQL_USERNAME=...
MYSQL_PASSWORD=...
TJU_LLM_API_URL=...
TJU_LLM_API_KEY=...
TJU_LLM_MODEL=tju-llm
RESOURCE_TOKEN_SECRET=至少32字节且长期稳定的随机值
```

Windows PowerShell 启动后端：

```powershell
cd apps/server
Get-Content ..\..\.env | ForEach-Object { if ($_ -match '^\s*([^#=]+)=(.*)$') { [Environment]::SetEnvironmentVariable($matches[1].Trim(), $matches[2].Trim(), 'Process') } }
mvn spring-boot:run
```

启动前端：

```bash
cd apps/web
npm install
npm run dev
```

本地访问：

```text
http://localhost:5173/
http://localhost:8080/api/health
http://localhost:8080/api/health/db
```

LLM 测试接口：

```powershell
$body = @{ message = '请只回复ok' } | ConvertTo-Json -Compress
Invoke-RestMethod -Uri 'http://localhost:8080/api/llm/test' -Method Post -ContentType 'application/json' -Body $body -TimeoutSec 90
```

## 7. GitHub Actions 部署

工作流文件：

```text
.github/workflows/deploy.yml
```

触发条件：

- 推送 GitHub `master` 分支。
- 手动 `workflow_dispatch`。

CI 阶段：

- 启动 MySQL 8.0 service。
- 使用 Java 17 执行后端测试。
- 使用 Node 20 构建前端。
- 校验 `docker-compose.prod.yml`。

CD 阶段：

- 通过 SSH 登录服务器。
- 使用 rsync 同步代码到 `/opt/agent2026-interview-agent`。
- 校验 `.env.prod` 的必填变量；首次缺少 `RESOURCE_TOKEN_SECRET` 时生成随机值并稳定写回，不在日志输出该值。
- 如果已有 MySQL 容器，先在 `backups/` 创建升级前数据库备份。
- 如果服务器 `.env.prod` 中 `ENABLE_TJU_VPN=true`，则组合使用：

```bash
docker-compose.prod.yml + docker-compose.vpn.yml
```

- 执行部署脚本，并等待 `server` 与 `web` 容器健康：

```bash
bash deploy/deploy-production.sh
```

GitHub Secrets 必需项：

```text
DEPLOY_HOST
DEPLOY_USER
DEPLOY_SSH_PRIVATE_KEY
```

可选项：

```text
DEPLOY_PORT
DEPLOY_PATH
```

## 8. 生产环境变量

服务器 `.env.prod` 至少需要：

```env
MYSQL_DATABASE=agent2026_interview_agent
MYSQL_USERNAME=agent2026_user
MYSQL_PASSWORD=replace_with_mysql_password
MYSQL_ROOT_PASSWORD=replace_with_mysql_root_password
RESOURCE_TOKEN_SECRET=replace_with_at_least_32_random_bytes

TJU_LLM_API_URL=https://ai.tju.edu.cn/api/agent2026/gitlab-42-agent2026-interview-agent/chat/completions
TJU_LLM_API_KEY=replace_with_tju_llm_api_key
TJU_LLM_MODEL=tju-llm

ENABLE_TJU_VPN=true
TJU_VPN_HOST=vpn.tju.edu.cn
TJU_VPN_USERNAME=replace_with_vpn_username
TJU_VPN_PASSWORD=replace_with_vpn_password

WEB_PORT=80
JAVA_OPTS=-Xms256m -Xmx512m
```

`RESOURCE_TOKEN_SECRET` 用于 HMAC 摘要项目资源访问令牌，必须满足以下规则：

- UTF-8 编码后至少 32 字节；
- 第一次部署后保持稳定，不能随普通发布轮换；
- 轮换后，之前签发的项目资源令牌将全部失效；
- 只保存在权限为 `600` 的 `.env.prod` 和安全备份中，不进入 Git 或日志。

### 8.1 Flyway 生产升级规则

- `apps/server/src/main/resources/db/migration/` 是数据库结构的唯一事实来源；`schema.sql` 和 `data.sql` 只作为旧资料保留，不再随应用启动执行，也不作为生产升级脚本。
- 自动部署会在启动新版本前，把运行中 MySQL 备份到部署目录的 `backups/predeploy-<UTC时间>.sql`，并自动清理超过 14 天的同类部署前备份；长期归档仍应复制到独立安全存储。
- MySQL DDL 不能依赖完整事务回滚。Flyway 迁移失败后不要反复重启应用，应先保留现场、核对已执行的 DDL，再按备份和 Flyway repair 流程恢复。
- 第一次在旧数据库启用 Flyway 前，必须确认旧库结构符合 V1 基线；`baseline-on-migrate` 不能代替结构核验。

如果启用 VPN sidecar，不需要手动设置：

```env
TJU_LLM_PROXY_HOST=ecvpn
TJU_LLM_PROXY_PORT=8888
```

这两个变量由 `docker-compose.vpn.yml` 注入到 `server` 容器。

## 9. VPN Sidecar 说明

生产服务器不能稳定直连学校 `ai.tju.edu.cn` 时，需要使用隔离的 EasyConnect sidecar。

相关文件：

```text
docker-compose.vpn.yml
deploy/ecvpn/Dockerfile
deploy/ecvpn/ecvpn-entrypoint.sh
deploy/ecvpn/vpn-connect.js
```

关键设计：

- EasyConnect 运行在 `agent2026-ecvpn` 容器内。
- `ecvpn` 容器暴露 HTTP proxy：`ecvpn:8888`。
- `server` 容器通过代理访问学校 API。
- 宿主机不直接运行 EasyConnect。
- 宿主机默认路由不被 VPN 接管，避免 SSH 和公网 Web 入口失联。

重要限制：

- 不要在宿主机上直接启动 EasyConnect。
- 如果宿主机曾启动过 EasyConnect，需要停止并禁用对应服务。
- `ecvpn` 必须能访问 `/dev/net/tun`。
- `ecvpn` 需要 privileged 或足够的网络能力。

当前 sidecar 里有一个关键路由修复：

- EasyConnect 会在容器内给 `tun0` 添加较宽的私网路由。
- 这些路由可能覆盖 Docker bridge 网段。
- 如果不修复，`server -> ecvpn:8888` 可能无法访问。
- `deploy/ecvpn/ecvpn-entrypoint.sh` 会删除和容器 `eth0` 网段冲突的 `tun0` 路由，并恢复 Docker bridge 路由。

这个逻辑不要轻易删除。

## 11. 常见问题

### 11.1 本地 PowerShell 中文乱码

项目文档是 UTF-8。PowerShell 终端可能因为编码显示乱码。读取中文文档时使用：

```powershell
Get-Content -Encoding UTF8 -Raw docs\项目交接与运维手册.md
```

### 11.2 生产 LLM 调用失败

优先检查：

1. `/api/health` 是否正常。
2. `/api/health/db` 是否正常。
3. `agent2026-ecvpn` 是否 healthy。
4. `server` 容器是否注入 `TJU_LLM_PROXY_HOST=ecvpn` 和 `TJU_LLM_PROXY_PORT=8888`。
5. `.env.prod` 中 `TJU_LLM_API_URL`、`TJU_LLM_API_KEY`、`TJU_LLM_MODEL` 是否存在。
6. VPN 账号是否仍可登录。

不要在日志里打印完整 Authorization header。

### 11.3 SSH 突然断开或公网访问异常

重点检查宿主机是否直接运行了 EasyConnect。

正确做法：

- 宿主机只跑 Docker、SSH、Web 入口。
- EasyConnect 只跑在 `agent2026-ecvpn` 容器中。
- 宿主机默认路由应继续走公网网卡，不应被 VPN 接管。

### 11.4 `server` 访问不到 `ecvpn:8888`

重点检查 `deploy/ecvpn/ecvpn-entrypoint.sh` 中的 Docker bridge 路由修复逻辑是否还在。

同时检查：

```bash
docker compose -f docker-compose.prod.yml -f docker-compose.vpn.yml --env-file .env.prod ps
docker compose -f docker-compose.prod.yml -f docker-compose.vpn.yml --env-file .env.prod logs --tail=120 ecvpn
```

### 11.5 GitHub Actions 没有部署

检查：

- 是否推送到了 GitHub `master`。
- GitHub Secrets 是否配置完整。
- CI 是否失败。
- `.env.prod` 是否存在于服务器部署目录。

如果只是推送学校 GitLab `main`，不会触发 GitHub Actions。

## 15. 接手者工作准则

继续开发时遵守：

- 先跑通现有链路，再改核心逻辑。
- 修改部署、VPN、LLM 客户端前先读 `deploy/README.md` 和本文档。
- Milvus 是增强层，不能让核心面试闭环依赖 Milvus 才能运行。
- 前端永远不直接持有学校 API Key。
- 任何新接口都应保持 `/api` 前缀，方便 Nginx 和 Vite 代理。
- 修改数据库表时新增 Flyway 迁移并更新相关文档；不要通过修改 `schema.sql` 升级生产数据库。
- 每次阶段性开发结束后更新 README 或本文档的当前状态。
